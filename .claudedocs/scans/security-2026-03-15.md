## CRITICAL Findings

### C1. Weak Password Hashing (SHA-256 without salt)
**File:** `src/lib/auth/index.ts:6,19`
**OWASP:** A02 - Cryptographic Failures

SHA-256 is used for password hashing. This is a fast hash — not a key derivation function. No salt is applied. Passwords can be cracked in seconds with GPU hardware or rainbow tables.

```typescript
// VULNERABLE - line 19
export function hashPassword(password: string): string {
  return createHash('sha256').update(password).digest('hex');
}
```

**Fix:** Replace with `bcryptjs` (already installed but unused):
```typescript
import bcrypt from 'bcryptjs';
export async function hashPassword(pw: string) { return bcrypt.hash(pw, 12); }
export async function verifyCredentials(email: string, pw: string) {
  return email === ADMIN_EMAIL && await bcrypt.compare(pw, ADMIN_PASSWORD_HASH);
}
```

### C2. Hardcoded Default Admin Credentials
**Files:** `src/lib/auth/index.ts:5-7`, `src/components/admin/LoginForm.tsx:119-131`
**OWASP:** A07 - Identification and Authentication Failures

Default credentials `admin@wbs.pl` / `admin123` are hardcoded as fallbacks AND displayed on the login page:

```typescript
// auth/index.ts
const ADMIN_EMAIL = process.env.ADMIN_EMAIL || 'admin@wbs.pl';
const ADMIN_PASSWORD_HASH = process.env.ADMIN_PASSWORD_HASH || createHash('sha256').update('admin123').digest('hex');
const SESSION_SECRET = process.env.SESSION_SECRET || 'wbs-default-session-secret-change-in-production';
```

LoginForm.tsx renders these credentials in a visible `<div>` with `<code>` tags.

**Fix:**
1. Remove all fallback defaults — throw error if env vars missing
2. Remove demo credentials display from LoginForm entirely (or gate behind `NODE_ENV === 'development'`)
3. Rotate all credentials immediately if code was ever publicly accessible

### C3. Missing CSRF Protection on All POST Endpoints
**Files:** `src/app/api/contact/route.ts`, `src/app/api/newsletter/route.ts`, `src/app/api/admin/login/route.ts`, `src/app/api/admin/logout/route.ts`
**OWASP:** A01 - Broken Access Control

No CSRF token validation on any POST endpoint. Malicious sites can trigger contact form submissions, newsletter signups, and admin logout on behalf of authenticated users.

**Fix:** Implement CSRF token middleware — generate token in layout, validate on all POST/PUT/DELETE endpoints.

---

### C4. Search Endpoint Exposes Staff PII (GDPR Risk)
**File:** `src/app/api/search/route.ts:45-159`
**OWASP:** A01 - Broken Access Control

The search API is unauthenticated and exposes staff email addresses, roles, and subjects. An attacker can enumerate the entire staff directory.

**Fix:** Remove `member.email` from search results. Consider requiring authentication for staff-related search results.

### C5. .env.local Contains Reversible Password Hash
**File:** `.env.local:8`

The stored hash `5e884898da28...` is the SHA-256 of `admin123` and is instantly reversible via rainbow table lookup.

**Fix:** Generate a proper bcrypt hash and update the environment variable. Ensure `.env.local` is in `.gitignore`.

### C6. Demo Credentials Visible in Production UI

Login page shows `admin@wbs.pl` / `admin123` in a styled box. Anyone visiting `/admin/login` gets full admin access.

## HIGH Findings

### H1. No Rate Limiting on Login Endpoint
**File:** `src/app/api/admin/login/route.ts:4-37`
**OWASP:** A07 - Identification and Authentication Failures

The login endpoint accepts unlimited authentication attempts. No lockout, no delay, no CAPTCHA. Contact form has rate limiting, but login does not.

**Fix:** Add rate limiting (5 attempts/15 min per IP), progressive delays, account lockout after repeated failures.

---

### H2. Session Token Forging Risk (Weak Secret)
**File:** `src/lib/auth/index.ts:7,22-24`

Default session secret is `'wbs-default-session-secret-change-in-production'`. With this known value, an attacker can forge valid session tokens. Token is Base64 payload + HMAC signature — readable in plain text (no encryption).

**Fix:** Require strong random secret via environment variable. Consider using `jose` JWT library instead of custom implementation.

---

### H3. Insufficient Input Validation on Contact Form
**File:** `src/app/api/contact/route.ts:26-28`
**OWASP:** A03 - Injection

Sanitization only strips `<>` characters. No length limits, no schema validation, no email injection protection. Zod is installed but not used.

```typescript
function sanitizeInput(input: string): string {
  return input.replace(/[<>]/g, '').trim(); // Insufficient
}
```

**Fix:** Use Zod schemas with strict field length limits and proper email validation.

---

### H4. Newsletter Endpoint Vulnerable to Email Injection
**File:** `src/app/api/newsletter/route.ts:45-53`

Email sanitization doesn't strip newlines (`\r\n`). An attacker could inject BCC headers: `attacker@evil.com\nBcc: victim@example.com`.

**Fix:** Validate with Zod `.email()`, implement double opt-in, add unsubscribe mechanism.

---

### H5. Missing Content Security Policy Header
**File:** `next.config.js:34-70`
**OWASP:** A05 - Security Misconfiguration

Other security headers are well configured (HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy), but CSP is completely absent.

**Fix:** Add CSP header:
```
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; frame-ancestors 'none';
```

---

### H6. Logout Endpoint Has No Auth Check
**File:** `src/app/api/admin/logout/route.ts`

Accepts POST from anyone with no session verification. Cross-site requests can force admin logout.

**Fix:** Verify session exists before deleting. Add CSRF token validation.

---

### H7. Only Single Admin User Supported
**File:** `src/lib/auth/index.ts:26-28`

Authentication supports exactly one hardcoded admin. No role-based access control, no audit trail of who made changes, no ability to revoke individual access.

**Fix:** Implement user database (even JSON-based) with roles and audit logging.

## MEDIUM Findings

### M1. In-Memory Rate Limiting (No Cleanup, Resets on Deploy)
**File:** `src/app/api/contact/route.ts:4-24`

Rate limit map grows indefinitely (memory leak). Resets on server restart. No coordination between instances.

**Fix:** Add periodic cleanup. Migrate to Redis-based rate limiting for production.

---

### M2. IP Address Spoofing for Rate Limit Bypass
**File:** `src/app/api/contact/route.ts:32`

`x-forwarded-for` header is trusted without validation. Attackers can spoof IPs to bypass rate limits.

**Fix:** Use trusted proxy configuration. Validate forwarded headers.

---

### M3. Newsletter Subscriber Data Stored Unencrypted
**File:** `src/app/api/newsletter/route.ts:5,25-30`

Email addresses stored in plain text JSON file on disk. GDPR requires encryption at rest for PII.

**Fix:** Encrypt email data at rest. Implement data retention policy.

---

### M4. Session Cookie Path Too Broad
**File:** `src/lib/auth/index.ts:43-49`

Cookie path is `/` (sent on every request) instead of `/admin/`. `sameSite` is `'lax'` instead of `'strict'` for admin panel.

**Fix:** Set `path: '/admin/'`, `sameSite: 'strict'`, `secure: true` always.

---

### M5. No Input Length Validation on API Payloads
**Files:** `src/app/api/newsletter/route.ts`, `src/app/api/contact/route.ts`

No Content-Length check. No max field length validation. DoS possible via enormous payloads.

**Fix:** Add body size limits and field max lengths via Zod validation.
